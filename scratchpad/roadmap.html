PHASE 1 / Event store & comfort

Merged · #188–#193

A foundation layer, not just features — the event store feeds the ship's log now and the sparklines later.

  • P1.1Audit logDone · #188–#190

    No server audit endpoint, so it's a client log: an append-only events table (SQLite — full history for stats, most-recent 1000 in memory) capturing emitted actions. #188 shipped the infra + first action set; #189 captured the rest (~20 actions); #190 folded in reconstructed server events (alerts + damage warnings).

  • P1.2Ship's log (narrated)Done · #188

    Captain's-log flow in the Missions pane (two-category root, mirroring Comms): dimmed timestamp, entities in accent, width-aware — truncated when compact, full when zoomed.

  • P1.3Naming ceremonyDone · #192

    No name-at-creation in the API → create/assemble then PATCH. Covers probes, drones (= probes), mannies (1-40 char, handle 409), containers (label 1-80). Themed lexicon proposed, cancelable to defaults.

  • P1.4Idle-manny cyclingDone · #193

    Flashy status-bar indicator ("N idle") + a "next idle" jump key. Cheap, independent.

  • P1.5Deuterium transferDone · #194

    API v86 catch-up: a manny ferries reserved deuterium to another fleet probe in the same sector. Two-step wizard (destination → amount) off the Mannies pane; same-sector rule is server-validated. Client version aligned to 86.0.0 (#196 → release #195).

PHASE 2 / Automation

Next

The big gameplay lever. Production queue, vim-style scripting and the rules engine all share one sequencing engine.

  • P2.1Production queue

    Queue of sequenced actions that fire as the previous one completes.

  • P2.2Vim-style scripting

    e.g. "detach a container → go → mine → bring it back". Same engine as the queue.

  • P2.3Rules engine (low prio)

    "if integrity < X then repair" — extends the same engine. Mind the blast radius: every step is a real API call.
